# Remnawave Grace Access

Worker для Remnawave, который выдает временный ограниченный VPN-доступ пользователям с истекшей или ограниченной подпиской.

Основной сценарий: пользователь забыл продлить VPN, доступ уже закончился или уперся в лимит, а Telegram в его регионе недоступен без VPN. Worker автоматически переводит такого пользователя в специальный squad, дает ему короткое окно доступа, например 3 дня и 1 GiB, чтобы он смог зайти в Telegram, открыть бота/канал/поддержку и продлить основную подписку.

Это не биллинг и не полноценная система продления. Это аварийный “grace access” для восстановления связи с пользователем.

## Что Делает

Раз в минуту worker сканирует пользователей Remnawave и обрабатывает статусы:

- `EXPIRED` — подписка истекла;
- `LIMITED` — пользователь ограничен, например из-за лимита трафика.

При обработке worker:

- выбирает служебный squad по статусу;
- заменяет все текущие internal squads пользователя на один служебный squad;
- переводит пользователя в `ACTIVE`;
- выставляет новый `expire_at` на `now_utc + EXTEND_DAYS`;
- ставит лимит трафика `TRAFFIC_LIMIT_BYTES`, по умолчанию 1 GiB;
- сбрасывает использованный трафик;
- записывает факт выдачи доступа в SQLite.

По умолчанию временный доступ выдается на 3 дня с лимитом 1 GiB.

## Логика Squads

Для разных причин блокировки можно использовать разные squads:

```text
EXPIRED -> TARGET_EXPIRED_SQUAD_UUID
LIMITED -> TARGET_LIMITED_SQUAD_UUID
```

Это удобно, если для истекших и ограниченных пользователей нужны разные маршруты, ноды, правила или тексты в панели.

Если отдельные squads не нужны, можно задать один общий `TARGET_SQUAD_UUID`. Он будет fallback для обоих статусов.

## Повторное Продление

Worker хранит состояние в SQLite и не раздает доступ бесконтрольно.

Если пользователь после выдачи временного доступа остается в одном из служебных squads, worker через `EXTEND_DAYS` дней снова выдаст новый период на `EXTEND_DAYS` дней и сбросит трафик.

Если пользователь израсходовал 1 GiB раньше 3 дней и Remnawave перевел его в `LIMITED`, worker не выдаст новый лимит сразу. Новый период и сброс трафика будут только после истечения `EXTEND_DAYS` с прошлого продления.

Если пользователь сам ушел из служебного squad в другой squad, worker больше не продлевает его автоматически. Снова он будет обработан только когда подписка опять попадет в `EXPIRED` или `LIMITED`.

Если пользователь уже находится в служебном squad, но worker раньше не выдавал ему временный доступ и в SQLite нет записи, worker не будет его продлевать. Это защищает от случайного подхвата пользователей, которых туда перенесли вручную.

## Как Работает Дата

Worker выставляет `expire_at` на:

```text
now_utc + EXTEND_DAYS
```

При `EXTEND_DAYS=3` пользователь получает ровно 3 дня от момента обработки. Старый `expire_at` не увеличивается и не переносится вперед, даже если он был в будущем.

## Настройка

Скопируйте пример env-файла:

```bash
cp .env.example .env
```

Минимально нужно заполнить:

```env
REMNAWAVE_API_BASE=https://remnawave.example.com/api
REMNAWAVE_API_TOKEN=put-token-here
TARGET_EXPIRED_SQUAD_UUID=e9534880-836d-41bc-9dc4-a453056ad5d1
TARGET_LIMITED_SQUAD_UUID=06e88dc4-6b6e-4db0-8de0-28c68ac12025
```

Основные параметры:

```env
TARGET_STATUSES=EXPIRED,LIMITED
EXTEND_DAYS=3
TRAFFIC_LIMIT_BYTES=1073741824
SCAN_INTERVAL_SECONDS=60
PAGE_SIZE=500
STATE_DB_PATH=./data/state.sqlite3
DRY_RUN=false
LOG_LEVEL=INFO
```

Права для Docker volume:

```env
PUID=1000
PGID=1000
```

На Linux-сервере значения можно посмотреть командой:

```bash
id
```

## Запуск

```bash
docker compose up -d --build
```

Логи:

```bash
docker compose logs -f worker
```

Остановка:

```bash
docker compose down
```

## Docker И SQLite

При `docker compose up` entrypoint сам:

- создает папку для SQLite по `STATE_DB_PATH`, например `./data`;
- выставляет владельца `PUID:PGID`;
- выставляет права `775`;
- запускает Python-процесс уже не от root.

Файл состояния хранит, кому и когда worker уже выдавал временный доступ. Без этого файла worker не сможет корректно отличать ручные действия от своих прошлых обработок.

## Проверочный Режим

Перед реальным запуском можно включить:

```env
DRY_RUN=true
```

В этом режиме worker пишет в лог, кого он бы обновил, но не отправляет `update_user`, не сбрасывает трафик и не записывает факт продления в SQLite.

Рекомендуемый порядок первого запуска:

1. Запустить с `DRY_RUN=true`.
2. Проверить логи и убедиться, что выбираются правильные пользователи.
3. Переключить `DRY_RUN=false`.
4. Проверить одного тестового пользователя в панели Remnawave.

## Что Проверить В Remnawave

После обработки тестового пользователя в панели должны измениться:

- status: `ACTIVE`;
- active internal squads: только выбранный служебный squad;
- expire date: примерно `now + EXTEND_DAYS`;
- traffic limit: 1 GiB или значение из `TRAFFIC_LIMIT_BYTES`;
- used traffic: сброшен.

## Ограничения

Worker не принимает платежи и не продлевает основную подписку. Он только дает короткий служебный доступ.

Worker не трогает пользователей, которые ушли из служебного squad вручную, пока они снова не попадут в `EXPIRED` или `LIMITED`.

Живое поведение API зависит от версии Remnawave. Локально проверена сериализация SDK `remnawave==2.7.1`, но перед production-запуском стоит сделать проверку на одном тестовом пользователе.
